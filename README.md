- Documentation folder

## Data and Authenticity Principle

For finance, quant, AI, and market-intelligence systems, Super follows this rule:

Build the core first. Licensed data scales the core later.

Near-term systems should use reliable, legal, explainable data sources such as:

- Official public datasets
- Exchange filings
- Annual reports
- Quarterly reports
- Corporate announcements
- Government and central-bank datasets
- SEC EDGAR filings
- FRED / public macro datasets
- Delayed or EOD data where permitted
- User-uploaded data
- Self-logged data
- Synthetic simulation data where clearly disclosed
- Low-cost vendor APIs where legally allowed

Projects should never pretend that public, delayed, synthetic, or self-logged data is the same as institutional real-time licensed data.

Authenticity comes from:

- Honest disclosure
- Strong architecture
- Correct methodology
- Reproducible outputs
- Validation
- Clear limitations
- Upgrade path to licensed data
- Professional documentation

## Engineering Philosophy

Super projects should be built with the mindset of serious engineering, not temporary scripting.

The preferred project pattern is:

1. Understand the theory
2. Build the simplest correct implementation
3. Add tests
4. Add examples
5. Add documentation
6. Add edge-case handling
7. Add professional structure
8. Add CI where useful
9. Add architecture notes
10. Add limitations and future integration notes
11. Lock the project only after final verification

## Guiding Rule

Every serious Super project should answer:
